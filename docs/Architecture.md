# Architecture (Draft)

## 프로젝트 구조
- `Nonogram.App`: WinUI 3 UI, 화면 상태, 사용자 상호작용, 앱 수명주기
- `Nonogram.Core`: 퍼즐 도메인 모델, Solver/Generator/Difficulty 엔진
- `Nonogram.Tests`: Core 중심 단위/통합 테스트, 핵심 시나리오 검증

## 핵심 모듈 책임

### Solver
- 퍼즐 상태 유효성 검사와 단계별 해법 적용을 담당한다.
- 해법 적용 기록(전략 로그)을 생성해 난이도 분석에 제공한다.

### Generator
- 유효한 퍼즐/정답 보드를 생성한다.
- 목표 난이도/크기 조건에 맞는 후보를 필터링한다.

### Difficulty
- Solver 로그와 탐색 비용을 바탕으로 난이도를 산정한다.
- 사용자 표시용 근거 요약(짧은 설명 텍스트)을 만든다.

### UI
- 사용자 입력, 렌더링, 상태 전환을 담당한다.
- Core 결과를 시각 상태(진행률, 경고, 완료)로 변환한다.

## 스레딩 원칙
- UI 스레드: 렌더링/입력/가벼운 상태 반영만 수행한다.
- 백그라운드: 생성/난이도 분석/대량 계산을 수행한다.
- 장시간 작업은 `CancellationToken`으로 취소 가능해야 한다.
- 진행 상태는 `IProgress` 또는 동등 메커니즘으로 UI에 전달한다.

## 경계 원칙
- `Nonogram.App -> Nonogram.Core` 단방향 의존을 유지한다.
- Core는 UI 타입에 의존하지 않는다.
- 테스트는 Core 계산 결과의 재현성과 결정성을 우선 검증한다.

## 핵심 결론
- 아키텍처의 핵심은 `Core 계산`과 `App 표현`의 분리다.
- 성능 리스크는 스레딩 경계로 제어한다.
- 난이도 설명 가능성은 Solver 로그 표준화에 달려 있다.
