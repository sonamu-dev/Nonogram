# Versioning (Draft)

## 결론 먼저
- `1.0.0.1` 형식은 Windows 앱(파일/패키지 버전)에서 일반적으로 맞는 방식이다.
- 다만 제품 버전 관리 기준은 `1.0.0`(SemVer)로 두고, 마지막 자리(`.1`)는 빌드 번호로 자동 관리하는 것이 좋다.

## 권장 버전 체계
- 제품 버전(사람 기준): `Major.Minor.Patch` 예) `1.2.3`
- 빌드/패키지 버전(시스템 기준): `Major.Minor.Patch.Revision` 예) `1.2.3.456`

## 빌드 번호 증가 정책
- 수동으로 매번 올리기: 실수 가능성이 높아 비권장
- 빌드할 때마다 자동 증가: CI 아티팩트 추적에 유리, 권장
- 릴리즈 때만 증가: 단순하지만 중간 빌드 추적성이 떨어짐

## 추천 운영 방식
- `Revision`(4번째 자리): CI 빌드마다 자동 증가
- `Patch/Minor/Major`: 릴리즈 시점에만 수동 결정
- 릴리즈 브랜치/태그 기준으로 제품 버전을 확정하고, 빌드 파이프라인이 `Revision`을 주입

## .NET/WinUI 적용 원칙(프로젝트 생성 후)
- `Version`: `Major.Minor.Patch` (예: `1.2.3`)
- `FileVersion`/패키지 버전: `Major.Minor.Patch.Revision`
- `AssemblyVersion`: 호환성 정책상 자주 바꾸지 않도록 고정 범위 사용(예: `1.2.0.0`)

## PR 체크 항목
- 이번 변경이 릴리즈 변경인지(= SemVer 증가 필요) 확인
- CI에서 `Revision` 자동 증가가 적용되는지 확인
- 사용자 노출 버전과 파일/패키지 버전이 일치 규칙을 따르는지 확인
